ExeWrap ZIP Installer Scaffold Plan

This plan turns the old app-builder two-layer 7zip SFX installer into a cleaner 1.x scaffold: ExeWrap becomes the first executable layer, the top archive becomes a stored ZIP with corrected offsets, and the second layer can keep the proven 7zip payload flow until the installer script itself is rewritten.

Branch: codex/app-builder-1x-rebuild PR: #62 Design issue: #53 ExeWrap: v2.0.0

Executive Shape

Recommendation: build this as silo scaffolding first. Do not make it the default release path until the ZIP overlay writer, 7zip payload builder, and installer top-layer assembly have focused tests and a Windows smoke test.

The current 1.x branch is not yet at the final install/uninstall workflow. That is a feature, not a blocker: the archive and executable layers can be made correct now, while the actual installer script remains a later integration point.

I use the repository name ExeWrap throughout this document; it refers to the WrapExe/ExeWrap first-layer executable wrapper discussed in the attached notes.

The clean split is: Python owns build-time artifact assembly, ExeWrap owns the outer executable launcher, PowerShell owns the small bootstrap script, Windows tar.exe owns first-layer extraction, and bundled 7z.exe owns second-layer payload extraction.

Current Facts

Fact Implication for app-builder
ExeWrap v2.0.0 publishes Windows x64, x86, and arm64 ZIP assets and its release notes list breaking config/template changes including numeric sentinels and stricter boolean handling. app-builder should vendor the 64-bit ExeWrap console launcher directly as package data. It is small enough that the installer path should not need a GitHub lookup, download cache, or digest check at build time.
ExeWrap supports an optional end marker after embedded config before unrelated suffix bytes. For an installer EXE with a trailing ZIP, app-builder must write [ExeWrap launcher][start marker][config][end marker][stored ZIP]. Existing Python venv launchers can omit the end marker because their config is still the final payload.
The attached experiment showed that .NET can read the hybrid EXE only when the ZIP central directory offsets are corrected for the full EXE/config preamble. PowerShell Expand-Archive works on the same bytes when the file is copied or renamed to .zip. The top layer must be written in place after the preamble, not built as a normal ZIP and naively concatenated afterward. Manual fallback can be documented as "rename to .zip and extract".
Python's zipfile append mode explicitly supports adding a ZIP archive to another file, such as an executable. The initial implementation can use the standard library for the top-layer ZIP writer, provided tests prove the central directory offsets remain compatible with .NET, PowerShell-after-rename, Python, and tar.exe.
Windows 10 added command-line tar and curl as default tools, while older Windows installations cannot be assumed to have tar.exe. The first self-extracting path can target modern Windows with tar.exe -xf installer.exe. The installer should still document the manual Windows fallback: rename the installer from .exe to .zip, then extract it.

Source references: ExeWrap release notes, Python zipfile docs, Microsoft Expand-Archive docs, and the Windows command-line tar announcement are linked at the end.

Legacy Pipeline

[7zSD.sfx + config.txt + installer.7z]        final installer.exe
        |
        | user runs exe
        v
temp top layer: install.bat, 7z.exe, 7z.dll, app.7z
        |
        | install.bat calls bundled 7z
        v
install dir: application payload, bin, shortcuts, uninstall script

The legacy implementation did useful work, but mixed too many responsibilities: project tools/ staging, release outputs, SFX assembly, archive remapping, script discovery, Windows shortcut creation, registry writes, and temp cleanup all leaked into one build path.

The 1.x plan should preserve the behavior that matters while deleting the worst habits: no project tools/releases staging, no ad hoc YAML peeking in batch files, no broad implicit script pickup, no writing temporary SFX helper files into the dist directory, and no shell-string archive logic when argv and Python APIs are available.

Target Pipeline

[ExeWrap-console.exe]
[config start marker]
{
  "command": [
    "powershell.exe",
    "-NoProfile",
    "-ExecutionPolicy",
    "Bypass",
    "-Command",
    "$ErrorActionPreference = 'Stop'; $exitCode = 0; $extractDir = Join-Path $env:TEMP ('app-builder-' + [guid]::NewGuid().ToString('N')); try { New-Item -ItemType Directory -LiteralPath $extractDir | Out-Null; tar.exe -xf '@{exe_path}' -C $extractDir; if ($LASTEXITCODE -ne 0) { $exitCode = $LASTEXITCODE; throw \"tar.exe failed with exit code $exitCode\" }; & (Join-Path $extractDir 'install.cmd'); $exitCode = $LASTEXITCODE } catch { if ($exitCode -eq 0) { $exitCode = 1 }; Write-Error $_ -ErrorAction Continue } finally { if (Test-Path -LiteralPath $extractDir) { Remove-Item -LiteralPath $extractDir -Recurse -Force -ErrorAction SilentlyContinue } }; exit $exitCode"
  ]
}
[config end marker]
[stored ZIP top layer with corrected central directory offsets]
top-layer ZIP contents
  install.cmd
  uninstall.cmd
  manifest.json
  payload archive (current 1.x zip; future 7z payload)
  bin/7z.exe / bin/7z.dll (future 7z payload pass)
  assets/...

The top ZIP is stored, not compressed. The current scaffold carries the existing 1.x payload zip; the eventual payload archive will be compressed by 7zip, so the top layer should stay fast, transparent, and friendly to Windows tools. Storing also makes the "rename to .zip and extract" fallback less surprising.

The ExeWrap config should create a unique directory under %TEMP%, extract itself with tar.exe -xf, call the extracted installer script, and clean up in a PowerShell finally block. In PowerShell, '@{exe_path}' is a valid literal string after ExeWrap template substitution, so app-builder does not need the more brittle JSON escape form \"@{exe_path}\" for this path. The exact bootstrap config should be generated by app-builder, not copied from user config.

If the production generator needs to be absolutely safe for paths containing apostrophes, it can pass @{exe_path} as a separate argv item after the -Command script and read it with param($installerExe). That is a refinement of the same PowerShell bootstrap approach, not a different direction.

Important boundary: ExeWrap is only the first-layer launcher. PowerShell is the first-layer bootstrap language. The second-layer installer behavior can remain 7zip based for now. A cleaner installer script can be planned separately after the artifact assembly is trustworthy.

ExeWrap Scope

The installer bootstrap should use one ExeWrap command that launches PowerShell. That keeps ExeWrap aligned with its core job: read embedded config, resolve templates, and launch one child argv. The process orchestration belongs in PowerShell because it already has try/finally, GUID generation, reliable path joining, and clear cleanup semantics.

Idea Decision Reason
commands Do not implement for this app-builder milestone. It is a legitimate future ExeWrap feature, but it turns ExeWrap into a small process orchestration engine. For the installer bootstrap, PowerShell gives us sequential commands without expanding ExeWrap.
finally_commands Do not implement for this app-builder milestone. PowerShell finally is exactly the cleanup mechanism needed here and preserves ExeWrap's simple one-child-process model.
random_hex(N) Not required for this milestone. PowerShell can use [guid]::NewGuid().ToString('N') directly while creating the temp extraction directory.

Keep commands, finally_commands, and random_hex(N) in the ExeWrap idea backlog. They may still be worthwhile if several non-PowerShell wrappers need orchestration, but they are not necessary for the app-builder installer.

Proposed Modules

Module Responsibility Cleanup from legacy
app_builder/exewrap.py Installer-specific ExeWrap package-data helper for the vendored x64 console launcher, config byte stamping, optional end-marker handling, and marker constants. Keeps installer EXE stamping offline and deterministic. The existing self-contained venv launcher path can be kept separate until we decide whether to vendor the windowed launcher as well.
app_builder/zip_overlay.py Writes stored ZIP entries onto an existing executable stream after the ExeWrap config end marker. Validates entry names, ZIP64 behavior, UTF-8 flags, and central directory compatibility. Replaces hand-built SFX concatenation with an explicit hybrid EXE/ZIP writer.
app_builder/sevenzip.py Owns 7zip invocation, quiet output filtering, listfile creation, compression settings, append mode, and temp-scoped resolution of 7z.exe plus 7z.dll. Retires the old file_pattern_7zip.py shape without losing the behaviors that mattered.
app_builder/archive_plan.py Builds a validated list of archive entries from include/exclude/remap rules. Detects duplicate destinations, path traversal, absolute-output destinations, and missing remap sources. Takes remapping out of 7zip-specific staging so it can also feed ZIP writers and tests.
app_builder/installer_bundle.py Creates the top-layer ZIP entries, installer manifest, and final ExeWrap installer EXE around the current payload archive. Later this is where the payload archive swaps to payload.7z. Owns the active first-layer installer artifact while the second-layer install behavior remains placeholder.
app_builder/install_scripts.py Renders the minimal current install.cmd and uninstall.cmd from structured manifest data. Later this becomes the home for a cleaner script rewrite. Avoids reviving functions.bat and the old implicit hook search.

7zip Base Behavior

The second layer should be a real 7zip payload builder, not a thin wrapper around the old code. It needs the legacy base behaviors, but expressed as small tested units.

Behavior Implementation plan Tests
Fairly silent compression Invoke 7zip with argv only. Disable progress where possible and stream-filter known banner/progress chatter while preserving warnings and errors. Do not hide non-zero exits. Fake-runner unit tests for command construction and output filtering. Windows integration test that creates a small archive without noisy progress output.
Locked-file support Detect files that are likely to upset 7zip on Windows, copy them to a temp staging tree when Python can still read them, and archive the staged copy at the original destination path. Use ctypes for Windows share-mode probing so pywin32 does not become a runtime dependency. Windows-only tests that hold a file open with restrictive sharing modes, prove the staging path is used, and prove temp staging is deleted. Include a readable-but-not-exclusive case because that is the legacy reason for this behavior.
Rename/remap Build an archive plan first: every source file maps to one POSIX destination path. Directory remaps are expanded before archive creation. Duplicate destinations fail before 7zip runs. Unit tests for direct file remap, directory remap, absolute source with required relative destination, duplicate destination rejection, and path traversal rejection.
Append mode Keep append support for adding generated system files such as version.txt or installer assets, but expose it through explicit method calls rather than toggling many parameters on one function. Archive-content test that creates an archive, appends a generated file, extracts it, and verifies all entries.
No project tool pollution Copy app-builder-owned 7zip assets through a temp directory when assembling artifacts. The only place bin/7z.exe appears is inside the generated top-layer installer ZIP for user-side extraction. Test that building an installer does not create or modify tools/, tools/releases, or any other project staging directory.

Implementation Phases

  1. Extract common ExeWrap support. Add app_builder/exewrap.py for installer stamping. Vendor one x64 ExeWrap console launcher as package data and add end-marker support for suffix bytes. Do not force the existing self-contained venv path through this helper until the windowed launcher question is decided.
  2. Add the ZIP overlay writer. Implement a stored ZIP writer that appends entries after an existing ExeWrap preamble. Use Python zipfile append mode initially, but verify central-directory offsets with real extraction tests.
  3. Replace the old archive planner. Promote include/exclude/remap into a neutral archive plan. This can extend the current fileset.py behavior, but destination validation belongs before any compressor runs.
  4. Implement the new 7zip runner. Add temp-scoped 7zip asset resolution, quiet command execution, locked-file staging, remap staging, and append support. This is the most intricate piece and should be reviewed like production code.
  5. Create installer bundle scaffolding. Build the current payload archive, manifest.json, installer scripts, top-layer stored ZIP, and the final ExeWrap installer EXE. Later this swaps the payload archive to payload.7z.
  6. Wire into build_release. Replace the placeholder installer zip now with the ExeWrap .exe carrying the existing payload zip and placeholder install script. The full user-side install behavior waits for the 7zip/script pass.
  7. Rewrite installer scripts later. The old batch scripts are full of historical baggage. The first integration can carry a minimal command script that calls 7zip and manifest-driven steps. A cleaner script engine can follow in a separate PR.

Worst Legacy Offences To Clean Now

Offence New rule
Project-root tools/ as a dumping ground for build intermediates and release staging. Use TemporaryDirectory for staging. Dist gets final artifacts only.
Concatenating SFX files and config files in the release directory. Build final EXE bytes through one artifact writer using temp staging and explicit markers.
One function handling globbing, remapping, locked files, copy mode, append mode, and 7zip execution. Split planning, staging, and compression into separate units with tests at each boundary.
Batch scripts scanning YAML or discovering hooks by filename. Installer behavior comes from manifest.json and explicit app_builder.yaml hook lists.
Hidden shell parsing and string command construction. Use argv internally. The ExeWrap bootstrap may use one generated PowerShell command string for structured extraction and cleanup.
Implicit reliance on bundled tools in mutable project folders. Resolve app-builder-owned tools from package resources into temp locations.

Deferred Work

  • A full rewrite of the installer and uninstaller command scripts can wait. This plan only requires a small manifest-driven bootstrap that preserves the second-layer extraction outcome.
  • Start Menu shortcut design can stay as currently modeled in 1.x until the installer script consumes it for real. The richer shortcut model from issue #53 can be a separate design pass.
  • Legacy support for R, Julia, Pandoc, Mintty, deploy-scripts, and repository dependencies should not be resurrected as part of this work.
  • Code signing and SmartScreen reputation matter for a public installer, but they do not block this scaffold. Keep the artifact writer deterministic enough that signing can be added later.

Open Decisions

Decision Recommendation
Should the first-layer self-extract command have fallbacks beyond tar.exe? Start with tar.exe for modern Windows. The fallback is a manual Windows support step: rename the installer to .zip and extract it. Add PowerShell/.NET fallback only if target clients prove they need automatic support.
Should the top ZIP be stored or compressed? Store it. The payload is already compressed, and stored entries improve speed and manual extraction friendliness.
Should 7zip be downloaded or packaged? For the first scaffold, reuse app-builder's existing bundled 7zip assets as package resources, copying them through temp staging. Later, replace with verified downloads if licensing/distribution review prefers that.
Should ExeWrap be downloaded or packaged? Package one 64-bit ExeWrap console launcher directly in app-builder. At roughly 300 kB, the simplicity is worth more than dynamic release resolution for this installer path.
Should app-builder keep the cmd.exe /C chain? No. Use one ExeWrap command that launches PowerShell with -NoProfile -ExecutionPolicy Bypass -Command. PowerShell is less brittle for temp directory creation, quoting, extraction, installer execution, and cleanup.
How should the temp extraction path become random? Use PowerShell [guid]::NewGuid().ToString('N') inside the bootstrap command. No ExeWrap template extension is required for this milestone.
Should ExeWrap add commands, finally_commands, or random_hex(N) now? No. They are reasonable future features, but PowerShell already supplies the orchestration needed by this installer while keeping ExeWrap focused on one templated child process.
Should this replace installer.zip immediately? Yes for the first-layer artifact. Emit an ExeWrap .exe that carries the existing payload zip and placeholder install script. The full user-side install waits for the later 7zip/script pass.
