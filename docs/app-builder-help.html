Overview

app-builder packages Windows-first applications from app_builder.yaml. It can prepare Python runtimes, run build steps, assemble payload files, create installers, create uninstallers, make Start Menu shortcuts, and publish artifacts to GitHub Releases.

The 1.x line is intentionally Python-focused. Features from old bin/... toolchains are retired except for Python. Non-Python tools still fit naturally through explicit build hooks.

Quick Start

Start inside a git repository:

app-builder init

Edit app_builder.yaml, then build local release artifacts:

app-builder release --version 0.1.0

Publish the same artifact set to GitHub Releases:

app-builder release-gh --version 0.1.0 --draft

Without --version, app-builder uses git describe --tags --always --dirty. If git cannot provide a version, it uses 0.0.0-dev.

Mental Model

Config

app_builder.yaml describes the app, install location, files, Python runtime, shortcuts, hooks, and app-builder version.

Build

Build hooks generate or verify assets. app-builder then collects files, applies remaps, and writes a payload archive.

Install

The installer extracts the payload, handles upgrades, runs install hooks, creates shortcuts, and writes uninstall records.

Release

The release command emits local artifacts. The GitHub release command uploads them through gh.exe.

Commands

Command What it does
app-builder --help Shows command help. This works even when no project config exists.
app-builder init [--force] Creates app_builder.yaml plus starter files under application-templates.
app-builder python Materializes only the configured bundled Python runtime.
app-builder deps Materializes configured Python environments without building release artifacts.
app-builder release [--version <version>] Builds payload, installer, and manifest files in installer.dist.
app-builder release-gh [--version <version>] [--draft | --no-draft] Builds artifacts and uploads or updates a GitHub Release through GitHub CLI.
app-builder 0.x <command> Runs the legacy 0.x compatibility bridge when installed.

Configuration

app_builder.yaml is the contract for a project. The loader is strict: unknown keys are rejected, legacy application.yaml shapes are rejected, and hook commands must be argv lists. The complete shape below is generated from the same schema metadata as app-builder init.

Start from the complete template when you are unsure. It shows every supported field, its default when omitted, and the short help text attached to that config option.

Command Values

Hook fields are list[list[string]]. Each command is an argv list. Use an explicit shell argv, such as [cmd.exe, /D, /C, ...], when shell behavior is required.

When a hook command starts with an existing .py file, app-builder runs it with the Python runtime configured for the project, preferring python_venv and then python_bundled. A hook such as [scripts/build.py] does not need python.exe on PATH. Use [python, scripts/build.py] only when you deliberately want the machine's own python.

Variables

app-builder resolves variables in string values before schema validation. This makes config values reusable without giving up strict structure: interpolation can build strings, but it cannot inject lists, mappings, or new config keys.

Interpolation is for ordinary config values, not for choosing the app-builder implementation. app_builder_version is read first by the command dispatcher, so use a literal value such as current, a tag, a branch, or a commit there.

Surface How to use it Values
Windows environment expansion Use percent-style variables in paths such as %localappdata%. %LOCALAPPDATA%, %APPDATA%, %USERPROFILE%, %TEMP%, plus normal Windows expansion.
Release version Use --version, or omit it and let git provide the version. git describe --tags --always --dirty, falling back to 0.0.0-dev.
Hook environment Read environment variables from build, install, and uninstall hooks. app_builder_name, app_builder_version, app_builder_install_directory, app_builder_project_root, app_builder_start_menu.

Config Interpolation

Use ${...} inside any YAML string. Values such as ${GIT.DESCRIBE}, ${ENV.BUILD_NUMBER}, or ${CONFIG.installer.name} can be inserted into other strings after the YAML is parsed and before the config schema is validated.

Missing variables are errors. CONFIG.* references resolve recursively, so the target can itself contain ${...}. Circular references are rejected. The final CONFIG.* target must be a string; app-builder will not splice lists or mappings into YAML.

Use percent-style Windows variables such as %localappdata% for install paths that must resolve on the user's machine. ${ENV.*} is resolved while building the release, so it bakes in the builder or CI environment.

For Windows paths, single-quoted YAML strings are usually easiest because backslashes stay literal. If you use double-quoted YAML strings for Windows paths, write backslashes as \\.

Variable Meaning
${ENV.NAME} Environment variable from the running process, for example ${ENV.BUILD_NUMBER}. Lookup is case-insensitive on Windows.
${GIT.DESCRIBE} git describe --tags --always --dirty, using app-builder's version fallback when needed.
${GIT.COMMIT} Full current commit hash.
${GIT.SHORT_COMMIT} Short current commit hash.
${GIT.BRANCH} Current branch name when available, otherwise an empty string.
${GIT.TAG} Exact tag at HEAD when available, otherwise an empty string.
${GIT.IS_DIRTY} Boolean-like string indicating whether the worktree has uncommitted changes.
${APP.VERSION} The release version after --version or git detection. This is the best value for artifact names and install metadata.
${CONFIG.path.to.value} Another resolved string value from the same config. List indexes can appear in the path, but the final target must be a string. 
installer:
  name: "MyApp ${APP.VERSION}"
  install_directory: '%localappdata%\Acme\${CONFIG.installer.name}'
  paths:
    include:
      - "build/${APP.VERSION}"
    remap:
      - ["README.md", "docs/${CONFIG.installer.name}.md"]

Payload Files

installer.paths controls what goes into the installed app. Includes and excludes are project-relative globs. Remaps move files or directories inside the payload. 

installer:
  paths:
    include:
      - README.md
      - src_py
      - native
      - build/generated
    exclude:
      - build/generated/*.tmp
      - "**/__pycache__"
    remap:
      - [README.md, docs/README.md]
      - [src_py, py]
      - [native, sources/native]
      - [build/generated/app.cmd, bin/app.cmd]

app-builder also writes version.txt into the payload. That file is not used as an installer identity marker.

Python Runtime

Python dependencies belong in pyproject.toml and are locked through Poetry. app-builder installs locked packages into Python runtimes configured for the project.

Mode Behavior
python_bundled enabled NuGet Python is materialized under bin/python. Main dependencies are installed into that runtime.
python_venv enabled with bundled Python A venv is created from bundled Python. Dev dependencies are installed into the venv.
python_venv enabled without bundled Python A self-contained NuGet-backed venv is created. Main and dev dependencies are installed into it.
Both disabled No Python runtime is materialized. .py hooks cannot auto-dispatch unless you call an explicit Python executable yourself.

Downloaded NuGet packages are cached under %TEMP%\app-builder-downloads.

Hooks

Hooks are always argv lists. Use an explicit shell when shell behavior is required. 

build_hooks:
  pre_dist:
    - [python, scripts/generate_assets.py]
  post_dist:
    - [python, scripts/verify_dist.py]

Relative hook paths resolve from the project root during build hooks and from the install directory during install hooks. If a hook command starts with an existing .py file, app-builder runs that file with the configured project Python, preferring python_venv and then python_bundled. That means [scripts/build.py] does not need python.exe on PATH. Use [python, scripts/build.py] only when you deliberately want the machine's own python. Existing .ps1 files run through PowerShell with execution policy bypass. Other commands are run directly as argv.

Installer

app-builder release produces:

  • <slug>-<version>.zip: the payload.
  • <slug>-<version>.7z: the payload when installer.payload_format: 7z.
  • <slug>-<version>-installer.exe: the self-extracting installer.
  • <slug>-<version>-manifest.json: the build manifest.

installer.payload_format selects the inner payload archive. zip is the default. 7z uses bundled 7-Zip files in the outer installer layer and preserves remapped paths plus Windows locked-file staging.

The installer extracts its outer ZIP layer to a random temp directory, runs the PowerShell installer, forwards all command-line arguments, and removes its temp extraction directory when finished. If an environment blocks running the exe, the manual fallback is: rename the installer to .zip, extract it, and run install.cmd.

Outer Installer Layout

install.cmd
bin/install.ps1
bin/uninstall.cmd       # when installer.add_uninstaller is true
bin/uninstall.ps1       # when installer.add_uninstaller is true
bin/7z.exe              # only when installer.payload_format: 7z
bin/7z.dll              # only when installer.payload_format: 7z
<app>-<version>.zip     # or .7z

Runtime Flags

Flag Behavior
--yes Bypass confirmation questions and skip the final close wait.
--no-wait Skip only the final close wait.

When installer.pause_on_exit is true and no bypass flag is supplied, the console closes after 30 seconds or when the user presses Enter. Other keys are ignored.

Existing Directory Rules

Existing directory Behavior
Current app-builder install for the same app Runs old pre_uninstall, moves the old install to a backup sibling, installs the new payload, and restores on failure.
Recognized legacy app-builder install for the same app Runs recognized legacy pre-uninstall hooks and removes the legacy install before installing the new payload.
Different app-builder app Refuses to overwrite.
Unknown directory Refuses to overwrite. Files such as version.txt, python-version.txt, and gitinformation.json are not install markers.

Uninstall

By default the installer writes bin\uninstall.cmd and bin\uninstall.ps1 into the installed app and creates an Uninstall <app name> Start Menu shortcut that points to the installed bin\uninstall.cmd.

The PowerShell uninstaller infers the install root from its own location, not from the manifest's install_directory. Uninstall runs pre_uninstall while the app is still present, removes Start Menu entries, stages allowed post_uninstall entrypoints, removes the install directory, then runs post_uninstall.

If a post_uninstall command points inside the install directory, its entrypoint must be a self-contained .cmd, .ps1, or .exe. Do not rely on helper files that will be deleted before the command runs.

Releases

Local releases are artifact generation only. GitHub releases use gh.exe. 

app-builder release --version 1.2.3
app-builder release-gh --version 1.2.3
app-builder release-gh --version 1.2.3 --draft

If the release already exists, app-builder uploads the current artifacts with --clobber. Otherwise it creates a new release with the payload archive, installer exe, and manifest JSON.

app-builder searches where.exe, PATH, and common GitHub CLI install locations. Install and authenticate GitHub CLI with: 

winget install --id GitHub.cli
gh auth login

app-builder Versions

app_builder_version controls which app-builder implementation runs for the project:

Value Behavior
missing, blank, or currentUse the installed 1.x app-builder.
v1.1.0, branch, or commitCheck out that ref into the managed version cache and run it.
0.x or v0.*Error. Use app-builder 0.x <command>.

Managed versions are cached under the installed app-builder directory in versions. The cache stores the requested ref, resolved commit, source repo, venv Python path, and dependency install command in version-manifest.json.

Legacy Projects

Old projects that still use application.yaml are not run automatically by 1.x. Use: 

app-builder 0.x <command>

The compatibility bridge is built from the old app-builder source with scripts/build_legacy_0x_bridge.py. It is intentionally isolated from the 1.x package.

Troubleshooting

No config found

Run app-builder init in a git repository, or move into a project that already contains app_builder.yaml.

Legacy config found

If the project still has application.yaml, use app-builder 0.x <command> or migrate to app_builder.yaml.

Python hook cannot run

Enable and materialize python_bundled or python_venv, or call an explicit Python executable in the hook argv.

Config interpolation failed

Check the exact variable name and the final value type. ${ENV.*} needs an environment variable that exists, ${GIT.*} needs a readable git repository, and ${CONFIG.*} must point at another string value without forming a cycle.

NuGet Python version missing

Use a broader prefix such as 3.12 or 3.12.*. app-builder suggests nearby available versions when selection fails.

GitHub release failed

Confirm gh.exe is installed and authenticated with gh auth status.

Installer refuses an existing directory

The directory is not recognized as a current or legacy app-builder install for the same app. Choose a clean install directory or remove the old directory deliberately.

Uninstall leaves diagnostics in temp

Failed asynchronous post-uninstall cleanup writes an error.txt in its temp staging directory so the failure is not lost.
